1. Executive Summary

Current Status

XIMA v1.1 is a functional assessment + hiring platform with GDPR launch compliance ("GO" verdict), a frozen psychometric baseline (Mean Cronbach's α 0.63–0.67), and a progressive 3-level challenge pipeline. The platform serves candidates, businesses, and mentors with AI-powered evaluation across 5 pillars.

Top Risks

  • AI Non-Determinism: No model pinning, no prompt versioning, no evidence trail for AI-evaluated open answers
  • Email Unreliability: Fire-and-forget email sends with no retry, no delivery tracking, no outbox pattern
  • No Observability: Structured console logs only — no persistent audit trail, no metrics, no alerting
  • Payment Gap: Membership tiers defined in code/DB but no payment integration — broken upgrade flows
  • Synthetic-Only Validation: Psychometric validity based on synthetic data only (N=0 real users for IRT)

v1.2 Objective

"Operational stability & auditability." Make XIMA enterprise-ready through deterministic AI governance, reliable event delivery, persistent audit trails, and honest feature gating.

4. Architecture Changes

4.1 AI Invocation Envelope

Every callAiGateway() call will persist a row to ai_invocations containing:

  • request_id (UUID, PK)
  • correlation_id (links to edge function request)
  • function_name (which edge function made the call)
  • model, provider, model_version (from gateway response)
  • prompt_hash (djb2 of system prompt — enables drift detection)
  • temperature, max_tokens (generation parameters)
  • input_length, output_length (no PII)
  • latency_ms, status, error_code
  • created_at

Why: Enables audit trail, drift detection, replay investigation, and cost tracking.

4.2 Assessment Evidence Ledger

New table assessment_open_answer_evidence linking each open-answer AI evaluation to its invocation:

  • open_response_id → assessment_open_responses.id
  • ai_invocation_id → ai_invocations.request_id
  • rubric_version (e.g., "v1.1")
  • raw_score, final_score, quality_label
  • red_flags (JSONB array)
  • guardrail_applied (boolean — was score capped by post-LLM rules?)

4.3 Audit Events

Lightweight event bus pattern via audit_events table:

  • id, actor_id, actor_type (user/system/admin)
  • action (enum: signup, assessment_start, assessment_complete, challenge_submit, data_export, account_delete)
  • entity_type, entity_id
  • metadata (JSONB, no PII)
  • created_at

4.4 Email Outbox

Idempotent email delivery via email_outbox table:

  • idempotency_key (unique: template + recipient_hash + context_hash)
  • status: queued → sending → sent | failed | retrying
  • attempts, max_attempts (default 3)
  • next_retry_at (exponential backoff)
  • provider_response (Resend response body)

6. Rollout Plan

6.1 Feature Flags

  • ENABLE_SELF_SERVE_PAYMENTS = false (disables Stripe checkout, shows Contact Sales)
  • ENABLE_L3_VIDEO = true or false (gates Level 3 invitations)
  • ENABLE_AI_INVOCATION_LOGGING = true (can be disabled in emergency)

6.2 Backward Compatibility

  • All new tables are additive — no existing table modifications
  • AI invocation logging is fire-and-forget — failures don't block the AI call
  • Email outbox is backwards-compatible — functions still send immediately, outbox tracks status
  • v1.1 assessment content is untouched — freeze guard enforces this

6.3 Rollback Strategy

  • Per-step rollback: Each step's migration can be reverted by dropping the new table
  • Code rollback: Git revert to pre-step commit
  • Feature flags: Instant disable without code deployment

7. Compliance & Legal Appendix

7.1 GDPR Decision-Support Disclaimer (Art. 22)

XIMA operates as a decision-support tool, not an automated decision-making system. Per GDPR Article 22:

  • All AI-generated signals (L1 mindset, L2 hard skills, CV analysis) are presented as qualitative indicators, not scores or rankings
  • Business users make final hiring decisions — XIMA does not auto-reject candidates
  • Candidates can exercise their right to object to profiling via profiling_opt_out flag
  • All AI evaluation functions check this flag before processing

7.2 Data Retention Recommendations

7.3 DSR Operations

  • Right of Access: data-export edge function aggregates 16 tables
  • Right to Deletion: delete-account edge function + delete_user_account RPC cascades across all tables
  • Right to Rectification: Profile edit UI + direct DB update via RLS
  • Right to Object to Profiling: profiling_opt_out flag blocks all AI analysis

7.4 Logging Boundaries

NEVER log: email addresses, names, answer text, CV content, chat messages

SAFE to log: user_id (UUID), correlation_id, function_name, latency_ms, status, field_key, content lengths, error codes

9. Change-Control SOP

For Future Releases (v1.3+)

  1. Proposal: Create stabilization plan document with risk heatmap and acceptance criteria
  2. Review: Engineering + Product sign-off on scope and priorities
  3. Execution: Step-by-step with stop points and rollback notes
  4. Validation: Each step verified before proceeding
  5. Freeze Update: If assessment content changes, bump version + regenerate hashes
  6. Documentation: Update plan HTML with completion status + changelog
  7. Release: Tag version, publish plan, update PLATFORM_VERSION constant

Assessment Content Change Protocol

  1. NEVER modify assessmentSets in locale files without version bump
  2. Run regenerateHashes() in browser console
  3. Update ASSESSMENT_FREEZE_HASHES with new values
  4. Bump ASSESSMENT_VERSION (e.g., "1.2")
  5. Rerun validation pipeline (Cronbach's α + discriminant validity)
  6. Document changes in release notes
