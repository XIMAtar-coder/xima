XIMA v1.2 — Validation Telemetry Playbook

Enterprise Psychometrics Monitoring · Version 1.0 · March 2026

1. What Is Collected

Core Principle: Zero PII

The psychometrics_telemetry table stores no user IDs, emails, names, or IP addresses. Each assessment run is identified by a one-way SHA-256 hash (run_id) derived from internal identifiers + a static salt. Re-identification is computationally infeasible.

What is NOT stored: raw text answers, content hashes, user IDs, email addresses, IP addresses, browser fingerprints.

2. Why It Is Collected

  • Psychometric Validity: Monitor pillar score distributions for ceiling/floor effects, skewness, and cross-pillar independence.
  • Assessment Health: Track completion rates, drop-off funnels, and error rates to catch regressions early.
  • Fairness Monitoring: Detect score distribution divergence across fields and locales (potential bias signals).
  • Operational Excellence: AI scoring latency, error rates, and red flag prevalence feed into SLA monitoring.

3. Retention & Access

Retention: 730 days (2 years). Each row has an expires_at timestamp computed automatically by trigger. Expired rows should be purged via scheduled job.

Access Control: RLS enforced. Only users with admin role in user_roles can read. No authenticated user can write — only service_role (via database triggers).

GDPR Status: Compliant — No personal data stored. No ability to re-identify individuals from telemetry alone.

8. Audit Trail (3-Event Sample)

-- When a user starts an assessment:
audit_events: { action: "telemetry.assessment_started" }  -- via metrics_daily increment
-- No audit_event row because start is lightweight; tracked via metrics_daily counter.

-- When assessment completes:
metrics_daily: { metric_name: "telemetry.assessment_completed", metric_value: +1 }

-- The telemetry row itself is the evidence:
psychometrics_telemetry: { run_id: "a3f8...", funnel_step: "result", ... }

9. GDPR Compliance Confirmation

  • ✅ No user_id stored — run_id is SHA-256(user_id || assessment_id || salt)
  • ✅ No email/name — not referenced anywhere in schema
  • ✅ No raw text — open answers stored only as numeric scores + quality counts
  • ✅ No content_hash reuse — evidence ledger hashes not copied to telemetry
  • ✅ Retention enforced — 730-day expiry via database trigger
  • ✅ Access restricted — RLS: admin-only read, service_role-only write
  • ✅ Re-identification impossible — salt is static but internal; no external mapping table exists
