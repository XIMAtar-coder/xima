XIMA v1.2 — Release Notes

Release Date: 2026-03-02 | Classification: Internal — Enterprise Engineering

1. Summary

XIMA v1.2 is an enterprise stabilization release focused on operational reliability, auditability, and compliance hardening. No new user-facing features were added. All changes strengthen the existing v1.1 platform for enterprise deployment.

2. What Changed

2.1 AI Governance New

  • All 8 AI-calling edge functions now persist invocation metadata to ai_invocation_log
  • SHA-256 prompt hashing enables drift detection without storing PII
  • Model version, temperature, max_tokens, and latency tracked per call
  • Admin-only replay endpoint: ai-invocation-replay

2.2 Open-Answer Evidence Ledger New

  • assessment_evidence_ledger links every AI-scored open answer to its invocation
  • Content integrity via SHA-256 hash (no raw text stored beyond assessment_open_responses)
  • Quality labels: excellent / good / acceptable / weak / flagged
  • Immutable link to AI invocation (ON DELETE RESTRICT)
  • 730-day automated retention policy

2.3 Email Reliability New

  • Transactional Email Outbox pattern with idempotency keys
  • Batch worker (process-email-outbox) with CAS locking
  • Exponential backoff: 30s → 60s → 120s → 240s → 480s → dead-letter
  • enqueue_email() SQL helper for server-side components

2.4 Audit Trail & Observability New

  • Immutable audit_events table (UPDATE/DELETE blocked by trigger)
  • metrics_daily for operational counters
  • Fire-and-forget emitAuditEvent() helper for edge functions
  • SQL-based alert thresholds (dead letters, error rates, completion rates)

2.5 Enterprise Contract Mode New

  • Self-serve payment flows replaced with "Contact Sales" CTA
  • business_entitlements table: plan tier, seats, feature flags
  • contact_sales_requests with audit event + email outbox integration
  • admin_set_business_plan RPC for manual entitlement management
  • check_business_feature() for server-side feature gating
  • New pages: /pricing, /contact-sales

2.6 L3 Video Hardening Hardened

  • GDPR: account deletion now removes challenge videos from storage
  • Audit trigger emits l3.video_uploaded on submission
  • L3 remains enabled — validated end-to-end (record, upload, store, playback, permissions)

2.7 Psychometric Telemetry New

  • Zero-PII psychometrics_telemetry table with SHA-256 anonymized run IDs
  • Automated triggers capture pillar scores, XIMAtar archetypes, funnel progression
  • 8 SQL reporting functions for weekly ops reports
  • 730-day automated retention
  • Playbook: XIMA_v1.2_Validation_Telemetry_Playbook.html

2.8 Freeze Guard Hard Lock Hardened

  • Production builds throw Error on null or mismatched assessment hashes — app will not start
  • Dev mode self-seals on first run, validates on subsequent runs
  • regenerateHashes() utility preserved for version bumps

3. Backward Compatibility

  • Assessment content: v1.1 frozen — zero changes to questions, options, or scoring
  • Database: All changes are additive (new tables/columns only). No existing table modifications.
  • API: All existing edge functions maintain their current signatures
  • Auth: No changes to authentication flows
  • Client: No breaking changes to candidate or business UX flows

4. Known Limitations

  • Freeze hashes not yet hard-coded: Self-sealing in dev mode; production will throw on null. Run regenerateHashes() in dev console and commit values before production deployment.
  • Single AI provider: All AI calls route through Lovable AI Gateway (Gemini). No fallback/circuit breaker.
  • Synthetic-only psychometrics: Cronbach's α validated on synthetic data only. Real-data IRT calibration requires N≥500 (telemetry infrastructure now in place).
  • Email provider: Requires RESEND_API_KEY secret. Worker gracefully skips if unconfigured.
  • No automated CI tests: Validation relies on SQL queries and manual verification.

5. Daily Health Check Queries

Query 1: Dead Letters & Failed Emails (24h)

SELECT status, count(*) as cnt,
  max(updated_at) as latest
FROM email_outbox
WHERE created_at > now() - interval '24 hours'
GROUP BY status
ORDER BY status;

Query 2: AI Scoring Error Rate (24h)

SELECT
  function_name,
  count(*) FILTER (WHERE status = 'success') as ok,
  count(*) FILTER (WHERE status = 'error') as errors,
  round(100.0 * count(*) FILTER (WHERE status = 'error') / NULLIF(count(*), 0), 1) as error_pct,
  round(avg(latency_ms)::numeric, 0) as avg_latency_ms
FROM ai_invocation_log
WHERE invoked_at > now() - interval '24 hours'
GROUP BY function_name
ORDER BY errors DESC;

Query 3: Assessment Completion Funnel (7d)

SELECT
  field_key,
  count(*) FILTER (WHERE funnel_step = 'started') as started,
  count(*) FILTER (WHERE funnel_step = 'mc_complete') as mc_done,
  count(*) FILTER (WHERE funnel_step = 'open_complete') as open_done,
  count(*) FILTER (WHERE funnel_step = 'result') as completed,
  round(100.0 * count(*) FILTER (WHERE funnel_step = 'result') / NULLIF(count(*) FILTER (WHERE funnel_step = 'started'), 0), 1) as completion_pct
FROM psychometrics_telemetry
WHERE completed_at > now() - interval '7 days'
GROUP BY field_key
ORDER BY started DESC;

Appendix B: Release Governance Model

B.1 Version Numbering

  • Major (X.0): Breaking changes to assessment methodology or data model
  • Minor (1.X): New features, stabilization, or compliance changes
  • Patch (1.2.X): Bug fixes, security patches (no schema changes)

B.2 Release Checklist

  1. All steps in stabilization plan marked COMPLETED
  2. Freeze guard hashes hard-coded and committed
  3. RLS audit passes for all enterprise tables
  4. Daily health queries return expected baselines
  5. Release notes HTML published
  6. PLATFORM_VERSION constant updated in codebase
  7. Stabilization plan marked with completion date

B.3 Hotfix Protocol

  1. Create branch from release tag
  2. Fix must not modify assessment content (freeze guard prevents this)
  3. Security fixes: emit audit event + notify admin
  4. Deploy, verify health queries, tag patch version
