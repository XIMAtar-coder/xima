2.2 MC Pillar Score (Guest Mode)

for each pillar_block (e.g., q1-5 → computational_power):
  scores = []
  for each question_id in block:
    if answer exists:
      raw = (answer_index + 1) * 2.5 + random(-0.75, +0.75)
      scores.push(clamp(raw, 0, 10))
  pillar_score = scores.length > 0 ? avg(scores) : 5.0

Range: [0, 10]
Source: XimatarAssessment.tsx:188-198

2.3 MC Pillar Score (Authenticated Mode)

Answer values stored in assessment_answers with pillar tag. Server-side trigger computes final scores when completed=true is set. The server computation logic is not in frontend code — it's triggered by the assessment_results.completed flag update.

2.4 Open Answer Scoring (Local Fallback)

total = length(0-20) + relevance(0-25) + structure(0-20) + specificity(0-20) + action(0-15)
Range: [0, 100]

length: ideal 80-250 words → 20pts; taper outside
relevance: field keyword hits × 2.5, cap 25
structure: sentence count + transitions + intro/conclusion markers
specificity: numbers + examples + long words
action: action verb count × 3, cap 15

Source: src/lib/scoring/openResponse.ts

2.5 Open Answer Scoring (AI Mode)

Edge function analyze-open-answer returns: score_total, score_breakdown (length/relevance/structure/specificity/action), steve_jobs_explanation, improvement_suggestions, cleaned_text.

2.6 Open→Pillar Blending

OPEN_WEIGHT = 0.10 (10% influence)

open1 → 60% creativity + 40% communication
open2 → 60% drive + 40% knowledge
computational_power → NOT affected by open answers

blended[pillar] = base[pillar] * (1 - 0.10) + open_contribution[pillar] * 0.10
Result clamped to [0, 100], rounded.

Source: src/lib/scoring/openResponse.ts blendOpenIntoPillars()

3. Drive Computation Logic

3.1 Storage

TableKey Columns
pillar_progress_snapshotsuser_id, source, occurred_at, pillar_scores (jsonb), metadata
profilesdrive_score (integer, cached), drive_updated_at

Valid sources: assessment_initial, mentor_session_completed, challenge_completed, assessment_retest

3.2 Single Snapshot Formula

pillars = [communication, knowledge, creativity, computational_power] (sorted ascending)
avg_weakest = (sorted[0] + sorted[1]) / 2
avg_strongest = (sorted[2] + sorted[3]) / 2

score = clamp(round(60 - (avg_strongest - avg_weakest) * 2), 0, 100)

Interpretation: Smaller gap between strongest and weakest → higher Drive score

3.3 Multiple Snapshots Formula (Growth Velocity)

1. Get baseline (earliest snapshot) and latest snapshot
2. Identify 2 weakest pillars at baseline
3. For each weak pillar:
   delta = latest_score - baseline_score
   velocity = delta / max(1, days_elapsed)
4. avg_velocity = mean(velocities of weak pillars)
5. target_velocity = 0.20 points/day
6. score = clamp(round(50 + (avg_velocity / 0.20) * 25), 0, 100)

Example: If weak pillars improved by 2 points over 10 days:
  velocity = 0.20, score = 50 + (0.20/0.20)*25 = 75

3.4 Trigger & Caching

Trigger trg_update_drive_on_snapshot fires on INSERT into pillar_progress_snapshots. Calls compute_drive_score(user_id) and updates profiles.drive_score and profiles.drive_updated_at.

5. ⚠️ Critical Findings

CRITICAL-001: Pillar Mapping Mismatch
The i18n category labels cycle through 5 pillars (q1=CompPower, q2=Comm, q3=Knowledge, q4=Creativity, q5=Drive, q6=CompPower...) but the code scores in sequential blocks (q1-5=CompPower, q6-10=Comm, q11-14=Knowledge, q15-18=Creativity, q19-21=Drive).

Impact: Only 4 of 21 questions (q1, q7, q13, q20) score into the pillar shown in the UI. All other answers are assigned to the wrong pillar.
Files: XimatarAssessment.tsx:358-371 vs en.json assessmentSets.*.questions.*.category
CRITICAL-002: Guest Scoring Uses Same Incorrect Mapping
Guest pillar computation (lines 202-207) uses the same sequential block mapping, affecting all guest users identically.
Files: XimatarAssessment.tsx:202-207
HIGH-001: All Fields Use Identical Code Mapping
The questionPillarMap is hardcoded and does not vary by assessmentSetKey. Despite each field having different question content with rotating pillar labels, the scoring always uses the same sequential blocks.
Files: XimatarAssessment.tsx:358-371
MEDIUM-001: Guest vs Authenticated XIMAtar Assignment Divergence
Guest users get XIMAtar via a simple if/else heuristic on top-2 pillars (11 branches). Authenticated users use Euclidean distance to 12 canonical profiles. Results may differ for the same answer set.
Files: XimatarAssessment.tsx:214-236 vs src/lib/ximatarTaxonomy.ts
LOW-001: No Max-Length on Open Answer Input
The Textarea has no maxLength attribute. Extremely long answers could be submitted.
Files: XimatarAssessment.tsx:561-565
